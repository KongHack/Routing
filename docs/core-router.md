# CoreRouter (`src/Core/CoreRouter.php`)

## Purpose

`GCWorld\Routing\Core\CoreRouter` is the central class responsible for handling incoming HTTP requests and directing them to the appropriate application logic. It acts as the primary engine for the routing process during the execution phase.

## Key Functionalities

### Initialization

*   **`getInstance(string $name = self::DEFAULT_NAME): RoutingInterface`**:
    *   Retrieves a singleton instance of `CoreRouter`. Multiple named instances can exist if different `$name` arguments are provided.
*   **`__construct(string $name)`**:
    *   The constructor is protected. Instances are typically created via `getInstance()`. It initializes the router with a specific name, allowing for multiple routing configurations if needed.

### Path Handling & Discovery

*   **`getPathInfo(): string`**:
    *   Determines the current request path from server variables like `$_SERVER['PATH_INFO']` or `$_SERVER['REQUEST_URI']`.
*   **`discoverRoute(?string $path = null): ?RouteDiscoveryData`**:
    *   Takes an optional path (or uses `getPathInfo()` if null) and utilizes `RouteDiscovery` to find a matching route definition.
    *   Returns a `RouteDiscoveryData` object if a route is found, otherwise `null`.

### Request Forwarding

*   **`forward(?string $path_info = null): void`**:
    *   This is the main method that orchestrates the routing process for an incoming request.
    *   It determines the request path (using `$path_info` or `getPathInfo()`).
    *   It uses `RouteDiscovery` (via `discoverRoute()`) to find the handler and any parameters for the given path.
    *   If a route is found:
        *   It handles session starting if the route requires it (`session` flag).
        *   It performs security checks (PEX) if configured.
        *   It instantiates the handler class associated with the route (e.g., `UserHandler`).
        *   It calls the appropriate method on the handler class based on the HTTP request method (e.g., `getXHR()`, `post()`, `get()`).
        *   It manages automatic page wrapping if enabled for the route and a `PageWrapper` is configured.
    *   If no route is found, it typically triggers a 404 hook.

### Reverse Routing

These methods allow you to generate URLs (paths) based on route names and parameters, which is crucial for creating links and redirects within your application without hardcoding URLs.

*   **`reverse(string $name, array $params = []): string`**:
    *   Generates a URL path for a given route `$name` and an array of `$params`.
    *   Throws `ReverseRouteNotFoundException` if the named route cannot be found or parameters don't match.
*   **`reverseMe(array $params = []): string`**:
    *   A convenience method to generate a URL for the *currently matched route* (`$this->getFoundRouteNameClean()`), applying new `$params` if provided.
*   **`reverseAll(string $name, array $params = []): array`**:
    *   Similar to `reverse()`, but returns the full route definition array instead of just the path string.

### Hooks

`CoreRouter` provides a hooking mechanism to allow developers to inject custom logic at various points in the routing lifecycle.

*   **`addHook(string $hookName, string $methodName): void`**:
    *   Registers a method (typically a static method on a class) to be called when a specific hook is fired. Hooks are namespaced by the router instance name.
*   **`fireHook(string $type, ?array $args = null): void`**:
    *   Triggers all registered callbacks for a given hook `$type`.
    *   Common hooks include: `before_request`, `after_request`, `404`, `403_pex`, `pre-session_start`, `post-session_start`, `before_handler`, `after_handler`, `before_request_method`, `after_request_method`, `after_output`.

### Key Properties/Configuration

These properties can be configured on the `CoreRouter` instance to modify its behavior:

*   **`setBase(string $base): void`**:
    *   Sets a base path that will be prepended to all URLs generated by `reverse()` and `reverseMe()`. For example, if your application is not at the root of the domain (e.g., `http://example.com/myapp/`), you would set `/myapp`.
*   **`setRoutePrefix(?string $prefix = null): void`**:
    *   Sets a prefix for all routes handled by this router instance. For example, if set to `/api`, a defined route `/users` will only match `/api/users`. This prefix is also considered during reverse routing.
*   **`setUserClassName(string $user): void`**:
    *   Defines the class name for user-related operations, primarily for permission checking (PEX). This class must implement `GCWorld\Interfaces\PEX`.
*   **`setPageWrapperName(string $name): void`**:
    *   Defines the class name for a page wrapper, which can automatically wrap content from handlers (e.g., apply a common HTML layout). This class should implement `GCWorld\Interfaces\PageWrapper`.
*   **`attachDebugger(Debugger $cDebugger): void`**:
    *   Attaches a `Debugger` instance to log route hits.
*   **`forceRoutes(array $routes): void`**:
    *   Allows manually setting an array of routes, bypassing the standard discovery from generated files.
*   **`attachRedisCache(\Redis $cRedis): void`**:
    *   Provides a Redis instance to `RouteDiscovery` for caching resolved routes.

## Interactions

*   **`RouteDiscovery`**: `CoreRouter` delegates the task of finding a route definition for a given path to `RouteDiscovery`.
*   **Handler Classes**: Once a route is matched, `CoreRouter` instantiates the corresponding handler class (e.g., specified in the `#[Route]` attribute or route definition array) and calls the relevant method on it (e.g., `get()`, `post()`, `getXHR()`). Handler classes typically implement interfaces like `HandlerInterface`, `AdvancedHandlerInterface`, or `JSONHandlerInterface` from the `GCWorld\Routing\Interfaces` namespace.
