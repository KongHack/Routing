# RouteDiscovery (`src/Core/RouteDiscovery.php`)

## Purpose

`GCWorld\Routing\Core\RouteDiscovery` is a crucial component in the route execution phase. Its primary responsibility is to take a given URL path and efficiently find the corresponding pre-processed route definition. It bridges the gap between an incoming HTTP request path and the application logic that should handle it.

## Key Functionalities

### Route Matching Execution

*   **`execute(string $path): ?RouteDiscoveryData`**:
    *   This is the main public method of the class.
    *   It takes the URL `$path` (after any `CoreRouter` prefixing) as input.
    *   It first attempts to load the route definition from a Redis cache if configured.
    *   If not found in cache (or Redis is not used), it proceeds to load and search through the generated `MasterRoute_*.php` files.
    *   It performs matching against the patterns defined in these files.
    *   If a match is found, it constructs and returns a `RouteDiscoveryData` object containing the matched route's pattern, its handler information, and any extracted URL parameters.
    *   If no match is found after checking all relevant sources, it returns `null`.

### Caching Mechanism

To optimize performance and reduce the overhead of file parsing and regex matching on every request, `RouteDiscovery` incorporates a caching layer using Redis.

*   **`setRedis(\Redis $redis): static`**:
    *   Allows injecting a Redis client instance into `RouteDiscovery`. When a Redis instance is present, `RouteDiscovery` will attempt to cache successfully resolved routes and retrieve them from Redis on subsequent requests for the same path.
*   **`purgeRedis(?string $path = null): static`**:
    *   Provides a way to clear cached routes from Redis.
    *   If a `$path` is provided, only the cache for that specific path is purged.
    *   If `$path` is `null`, the entire cache related to the current router instance name is purged. This is useful when route definitions change and the cache needs to be invalidated.
*   **Cache Key Strategy**:
    *   Cache keys are typically constructed using the router instance name and the request path to ensure that different router instances or paths don't have cache collisions.

### Route Loading Strategy

`RouteDiscovery` intelligently loads the generated route files to find a match.

*   **Generated Files**: It relies on the PHP files generated by `Processor.php` (e.g., `MasterRoute_YourMaster.php`, `MasterRoute_MISC.php`) located in the configured `Generated/<RouterName>/` directory.
*   **Master Route Files**: It first tries to identify a "master" segment from the path (usually the first part of the URL, e.g., `/admin/...` would look for `MasterRoute_ADMIN.php`).
*   **MISC Route File**: If a specific master route file doesn't exist or a match isn't found within it, `RouteDiscovery` falls back to checking `MasterRoute_MISC.php`, which contains routes not grouped under a specific master segment.
*   **Forced Routes**: If routes have been forced via `CoreRouter::forceRoutes()` and subsequently passed to `RouteDiscovery::forceRoutes()`, these will be used instead of loading from generated files.

### Path Matching

`RouteDiscovery` employs several strategies to match the incoming path against the loaded route patterns:

*   **Direct Match**: It first checks for an exact match between the request path and a defined route pattern.
*   **Regex Matching**: If no direct match is found, it iterates through routes that contain placeholders (e.g., `/:id/`, `/:slug/`).
    *   It uses a predefined set of `ROUTING_TOKENS` (defined in `ConstantsInterface`) to convert these placeholder patterns into regular expressions. For example, `:id` might translate to `([0-9]+)`, and `:slug` to `([a-zA-Z0-9_-]+)`.
    *   If a regex pattern matches the path, the captured values (like the actual ID or slug) are extracted.

## Return Value: `RouteDiscoveryData`

When `execute()` successfully matches a route, it returns an instance of `GCWorld\Routing\Core\RouteDiscoveryData`. This object encapsulates the information about the matched route:

*   **`pattern: string`**:
    *   The raw route pattern that was matched (e.g., `/user/:id`).
*   **`handler: array<string, mixed>`**:
    *   An associative array containing the handler information for the matched route. This array is typically derived from the `Route::getRouteArray()` method and includes details like:
        *   `class`: The fully qualified class name of the handler.
        *   `name`: The name of the route.
        *   `session`: Boolean indicating if a session is required.
        *   `autoWrapper`: Boolean indicating if page wrapping should be applied.
        *   `pexCheck`, `pexCheckAny`, etc.: Permission check strings.
        *   `preArgs`, `postArgs`: Arguments to be prepended/appended to the handler method.
        *   `meta`: Meta-information for the route.
*   **`matches: array<array-key, string|int>`**:
    *   An array containing the values extracted from the URL path that correspond to any placeholders in the matched pattern. For example, if the pattern is `/user/:id` and the path is `/user/123`, `matches` would typically be `['123']` (or include the full path match as the first element depending on regex flags, like `['/user/123', '123']`).
